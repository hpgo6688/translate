# 整体架构设计

> 配套文档:[模块拆分](modules.md) · [技术选型](tech-stack.md) · [服务端 API](api.md) · [功能需求](function.md)

## 1. 设计原则

1. **客户端优先**:浏览器扩展能本地完成的(网页翻译、缓存、划词、配置)绝不上服务端,降低延迟与运营成本。
2. **服务端按需**:只承载客户端做不了或做不好的能力 —— Key 托管、长文档异步处理、跨设备同步、共享缓存、计费。
3. **协议中立**:对客户端暴露统一翻译协议,内部各 Provider(Google/DeepL/OpenAI/Claude/Gemini/...)可热插拔。
4. **优雅降级**:服务端不可用时,扩展退化为「自带 Key 直连模式」,核心翻译能力不丢失。
5. **同构部署**:SaaS 与自部署共用同一份服务端代码,差异由配置开关控制。

## 2. 部署形态

| 形态 | 用户群 | Key 来源 | 服务端 | 收费 |
|---|---|---|---|---|
| **Free / 自部署** | 个人极客、企业内网 | 用户自带 | 可选(自部署可解锁 PDF 等高级功能) | 免费 |
| **SaaS / Pro** | 大众用户 | 平台托管 | 必需 | 订阅 + 按量 |

扩展启动时按登录状态自动选择路径,未登录走 Free,登录走 SaaS。两种模式下网页翻译的 UX 完全一致。

## 4. 客户端 / 服务端 职责边界

| 能力 | 客户端 | 服务端 |
|---|---|---|
| 网页 DOM 解析与译文注入 | ✅ | ❌ |
| 段落级本地缓存(IndexedDB) | ✅ | — |
| 共享缓存(跨用户同段落复用) | — | ✅ (SaaS) |
| 同步翻译请求 | Free 直连 / SaaS 转发 | ✅ SaaS |
| API Key 加密存储 | ✅ Free | ✅ SaaS |
| 用量统计 | 本地展示 | 权威数据源 |
| 长文档 PDF / EPUB 翻译 | 上传 + 展示进度 | ✅ 异步 Job |
| OCR / 视频转录 | — | ✅ |
| 术语表 / 配置同步 | 本地 | ✅ |
| 订阅 / 计费 | — | ✅ |

## 5. 关键数据流

### 5.1 网页双语翻译(同步流式)

1. content script 扫描 DOM,识别可翻译段落,生成 `[{id, text}]`(id 是节点稳定 hash)。
2. 批量发往 service worker,后者按域名规则、本地缓存、配额做预处理。
3. 命中缓存的直接返回;未命中的合并请求发往 gateway(SaaS)或 provider SDK(Free)。
4. gateway 通过 SSE 逐段 stream 回客户端。
5. content script 按段落 id 找回 DOM 节点,注入双语容器。

### 5.2 PDF / EPUB 长文档翻译(异步)

1. 客户端上传文件到 `POST /v1/jobs/document`,gateway 落对象存储 + 创建 job + 入队。
2. LangGraph worker 消费:解析 → 分章节 → 顺序翻译并维护术语一致性 → 重排版输出。
3. 客户端通过 SSE 订阅 `GET /v1/jobs/{id}/events` 获取进度,完成后下载结果。

### 5.3 字幕实时翻译

1. content script 监听视频字幕轨道变化,捕获 cue。
2. 通过 SSE 长连接持续发送给 gateway。
3. gateway 做小批量合并(防抖 200ms) + provider 调用。
4. 译文回流后注入 `<video>` 上方的双语字幕容器。

## 6. 何时引入 LangGraph

LangGraph 的价值在「多步骤有状态工作流」。只用于下列场景,**不要套在普通翻译请求上**:

| 用 LangGraph | 不用 LangGraph |
|---|---|
| 长文档分章节翻译 + 术语一致性 | 单段落翻译 |
| 学术论文:抽术语 → 译 → 自校对 → 反向回译 | 划词 / 悬停翻译 |
| OCR 流水线:预处理 → 识别 → 版面 → 译 → 渲染 | 字幕实时翻译 |
| 多模型并行 → 评分 → 选优 | 输入框翻译 |

判定标准:**有状态、多步骤、可能失败需重试某步、需要分支/合流** ⇒ 上 LangGraph;否则直接 provider SDK。

## 7. 非目标

- 不自研翻译模型,只做编排
- 不自研 PDF 渲染,用 PDF.js + 自己做版面映射
- 不做完整 OCR 引擎,接 PaddleOCR / Tesseract 或商用 API
- 不做浏览器内核改造,纯 Web Extension(MV3)
- 第一阶段不做移动端 / 桌面客户端

## 8. 演进路线与服务端引入时机

| 阶段 | 服务端 | 关键里程碑 |
|---|---|---|
| **P0 MVP** | 不需要 | 纯客户端跑通网页翻译 + 本地缓存 + Google/DeepL |
| **P1** | 引入 Go gateway | SaaS 上线、Key 托管、共享缓存、计费、AI 模型接入 |
| **P2** | 引入 LangGraph worker | PDF/EPUB 异步翻译、术语一致性 |
| **P3** | 扩展 worker 能力 | OCR、多模型投票、Agentic 学术翻译、云同步 |

## 9. 关键风险与对策

| 风险 | 对策 |
|---|---|
| Provider 限流 / 不稳定 | gateway 做指数退避 + 自动降级链(Claude → OpenAI → Google) |
| 长文档 worker 超时 | LangGraph checkpoint 落 Postgres,允许断点续译 |
| 共享缓存命中率低 | 段落 hash 用「归一化文本」(去空白/标点)而非原文 |
| MV3 service worker 频繁休眠 | 重要状态写 IndexedDB,不依赖内存常驻 |
| 浏览器跨域/CSP 限制 | 所有 provider 调用都从 background 发起,content script 只发消息 |
